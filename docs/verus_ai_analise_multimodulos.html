🛠️ Sumário de Correções

🔴 P0 — Financeiro: Frontend x Backend inconsistente

Módulo: cases / frontend | Evidência: urls.py + 4 arquivos frontend

Backend desativou URLs do financeiro (# comentado no urls.py), mas o frontend ainda tem 4+ páginas chamando `/api/v1/processos/financeiro/*`. Todas essas chamadas retornarão 404 silenciosamente.

Necessita correção urgente

🟡 P1 — Código morto: process_document_task nunca chamada

Módulo: kb | Evidência: tasks.py + grep .delay()

A task `process_document_task` em `apps/kb/tasks.py` é definida com `@shared_task` mas NUNCA é invocada via `.delay()` ou `.apply_async()` em lugar nenhum do código. Processamento de documentos KB é apenas síncrono.

Remover ou implementar chamada

🟡 P1 — Duplicação de pipeline RAG

Módulo: rag / intelligent_assistant | Evidência: models.py + knowledge_base_service

O app `rag` define modelos RAGQuery e RAGContext que referenciam `documents.Document`. Paralelamente, o `intelligent_assistant` tem seu próprio pipeline RAG via KnowledgeBaseService + pgvector. Duas implementações RAG distintas.

Consolidar pipelines

🔵 P2 — Módulo collaboration isolado (silo potencial)

Módulo: collaboration | Evidência: grafo de dependências

Nenhum outro app do sistema referencia models do `collaboration` via FK. É um módulo funcional mas sem integração com o resto do sistema.

Monitorar uso real

🔵 P2 — Duplicação de conceito "Documento"

Módulo: documents / intelligent_assistant | Evidência: models.py

O app `documents` define DocumentGenerator e Document. O `intelligent_assistant` define GeneratedDocument e IntelligentSession. Conceitos sobrepostos de "documento" em apps diferentes.

Avaliar unificação futura

🔵 P2 — Testes concentrados em cases app

Módulo: todos | Evidência: pytest.ini + test directories

Casos de teste reais (APITestCase) estão concentrados no app `cases`. Apps críticos como intelligent_assistant, workflow_execution, simulations têm diretórios de teste mas cobertura não confirmada.

Expandir cobertura

📋 Fases Detalhadas

1. Autenticação & Multi-tenancy

Frontend Backend DB

Módulos: accounts, organization

Entrada: Credenciais do usuário

Processamento: JWT (access 1h + refresh 7d), single-flight refresh, role hierarchy (11 níveis)

Saída: Token JWT com claims de organ/role

Decisões: Token expirado? → Refresh / Redirect login

Erros: Refresh blacklisted mas não expirado? Single-flight trata race condition

2. Gestão de Processos (cases)

Frontend Backend DB

Módulos: cases, accounts, organization

Endpoints: ~120+ (CRUD + prazos + tarefas + documentos + contratos + NFS-e + CRM + KPI + OCR + petição IA)

Modelos: LegalCase, Client, LegalDeadline, CaseTask, CaseDocument, LegalContract, Lead, RiskAssessment, InvoiceNFSe

Integrações: active_flow → workflow_execution, linked_document → intelligent_assistant

Decisões: active_flow exists? → Serializa info do workflow

3. Geração de Documento com IA

Backend Frontend DB

Módulos: intelligent_assistant, core, kb, templates, forms

Endpoints: ~60+ (sessions, blueprints, agents, generation, export)

Fluxo: Blueprint → FormTemplate → Upload → Geração SSE (multi-provider) → Revisão → Exportação (PDF/DOCX/ODT) → R2

LLM Providers: Anthropic (default), OpenAI, IBM WatsonX — configuráveis via core.LLMProvider

RAG: SectionAgentConfig ↔ KnowledgeBase ↔ pgvector embeddings

Decisões: Provider primário falha? → Retry 3x com backoff; Provider selecionado por blueprint ou global?

⚠️ save_pdf/docx_to_r2_task sem fallback se R2 offline

4. Workflow BPMN

Backend DB

Módulos: workflow_definition, workflow_execution, accounts, organization

Fluxo: Template → Publicar → Instanciar → Tasks → Gateways (exclusive/parallel) → Completar

Modelos: FlowTemplate, FlowNode, FlowEdge, FlowVersion, FlowInstance, TaskInstance, TaskRequest, ExecutionEvent

Integrações: cases.LegalCase.active_flow (O2O), signal sync_case_flow_status limpa ao completar

Decisões: exclusive_gateway → usuário escolhe caminho; parallel_gateway → tasks simultâneas

Auditoria: ExecutionEvent imutável para cada transição

5. Copilot & Tasks Periódicas

Backend Frontend

Módulos: copilot, cases, accounts, intelligent_assistant

Tasks: 7 Celery Beat tasks (check_upcoming_deadlines/h, analyze_idle_cases/diário, check_pending_documents/diário, process_user_reminders/5min, nightly_case_analysis/meia-noite, sync_user_knowledge_bases/2h, cleanup_stuck_simulations/2h)

Fluxo: Session → Chat SSE → RAG → LLM → Histórico → Compartilhar (share_code)

6. Simulações (18 tipos)

Backend DB Frontend

Tipos: jury, judge, stf, stj, tst, trt, tre, tse, jec, jecrim, acordao, eleitoral, trabalho, militar, turma_recursal, stm, tre, trt

Serviços: JurySimulationService, JudgeSimulationService, STFSimulationService, etc. — cada tipo tem service próprio

Fluxo: Criar simulação → Upload docs → SSE stream (debate IA multi-fase) → Veredicto → PDF export

Decisão: Simulation.status = running/completed/failed; cleanup_stuck marca como failed após 2h
